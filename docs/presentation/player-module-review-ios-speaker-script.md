## 사용법

각 슬라이드는 `주장 -> 화면 -> 말할 내용 -> 전환` 순서로 말한다.
발표 중에는 화면 섹션을 먼저 짚고, 말할 내용은 3-5문장 안에서 끝낸다.

## 01. 왜 지금 플레이어 모듈화인가

**주장**  
플레이어는 서비스별 구현물이 아니라 팀 공통 자산이 되어야 합니다.

**화면**  
1. 고등, 유초등, 해외사업부 카드를 차례로 짚습니다.
2. 마지막 GOAL 카드에서 공통 모듈이 목적임을 말합니다.
3. 하단 takeaway로 결론을 고정합니다.

**말할 내용**  
지금 문제는 새 플레이어 하나가 아니라, 같은 재생 문제를 서비스마다 다시 푸는 구조입니다. 재생, 제스처, 자막, 배속, DRM, 다운로드는 대부분 공통 문제입니다. 그래서 출발점은 “여러 서비스가 같은 문제를 다시 풀지 않게 하자”입니다.

**전환**  
그 반복 구현이 실제로 어떤 비용으로 남는지 보겠습니다.

---

## 02. 현재 구조의 비용

**주장**  
반복 구현은 기능 개발보다 유지보수와 검증 비용을 키웁니다.

**화면**  
1. UX 카드는 제스처, 자막, 배속 반복 구현을 뜻합니다.
2. KOLLUS 카드는 SDK가 시뮬레이터에서 실행되지 않는다는 제약입니다.
3. SDK와 PRODUCT 카드는 버전 대응과 경험 분산 비용입니다.

**말할 내용**  
여기서 핵심은 기능이 많다는 것이 아니라 같은 판단을 서비스마다 반복한다는 점입니다. 특히 Kollus 경로는 SDK 자체가 시뮬레이터에서 실행되지 않기 때문에 실기기 검증이 필요합니다. 공통 경계가 없으면 SDK 버전 대응과 사용자 경험 차이도 계속 서비스별 비용으로 남습니다.

**전환**  
이 비용이 실제 코드 표면에 어떻게 남아 있는지 보겠습니다.

---

## 03. 반복 비용의 근거

**주장**  
모듈화는 추상 정리가 아니라 이미 분산된 책임을 한 경계로 모으는 작업입니다.

**화면**  
1. 72는 레거시 플레이어 영역의 Objective-C 파일 수입니다.
2. 82는 Kollus 관련 SDK 접점 검색 결과입니다.
3. QA 카드는 Kollus 경로 전체가 실기기 검증 대상임을 뜻합니다.

**말할 내용**  
이 숫자는 “느낌상 복잡하다”가 아니라 코드 표면에 이미 비용이 남아 있다는 근거입니다. SDK 타입, storage, delegate, 다운로드 API가 여러 곳에 흩어지면 변경과 검증도 흩어집니다. 그래서 공통 경계가 필요합니다.

**전환**  
이제 목표 구조를 보겠습니다.

## 04. 목표는 공통 재사용 모듈

**주장**  
공통화할 것은 재생 구조이고, 달라질 것은 Host가 주입합니다.

**화면**  
1. 왼쪽 묶음은 이 모듈을 소비하는 서비스 Host입니다.
2. 가운데 묶음은 여러 서비스가 재사용할 공통 재생 모듈입니다.
3. 오른쪽 묶음은 서비스별로 달라질 Host 주입점입니다.
4. 화살표는 Host가 공통 모듈을 재사용하고, 필요한 차이만 주입한다는 흐름입니다.

**말할 내용**  
모든 서비스가 같은 화면을 쓰자는 뜻이 아닙니다. 재생 명령, 상태, 정책 협상, 엔진 계약은 공통화하고, 서비스별 UI나 정책은 주입합니다. 공통과 커스텀의 경계를 나누는 것이 목표입니다.

**전환**  
구조 이야기에 들어가기 전에 용어를 맞추고 가겠습니다.

---

## 05. 오늘의 핵심 용어

**주장**  
Host, Core, Engine, Skin, Command/State만 잡으면 이후 구조를 따라갈 수 있습니다.

**화면**  
1. Host는 서비스 앱입니다.
2. Core는 상태 소유자입니다.
3. Engine은 실제 SDK를 호출하는 번역기입니다.
4. Skin은 화면 조립이고, Command/State는 공통 언어입니다.

**말할 내용**  
이후 슬라이드는 이 다섯 단어가 어디에서 만나고 어디에서 분리되는지 설명합니다. 모듈화가 낯설어도 이 단어만 따라가면 전체 흐름을 놓치지 않을 수 있습니다.

**전환**  
이제 단순 wrapper와 공통 모듈의 차이를 코드로 보겠습니다.

---

## 06. Wrapper가 새는 순간

**주장**  
얇은 wrapper는 SDK 호출을 숨겨도 SDK의 사고방식까지 숨기지는 못합니다.

**화면**  
1. 노란색 KollusPlayerView를 봅니다. 화면 계층이 SDK 타입을 직접 압니다.
2. 분홍색 setStorage, setDelegate를 봅니다. SDK 조립 규칙도 화면에 올라옵니다.
3. 마지막 상태 enum을 봅니다. SDK 콜백이 화면 상태를 직접 움직입니다.

**말할 내용**  
이런 구조에서는 wrapper를 만들어도 Host가 여전히 SDK 타입, delegate, callback 순서를 알아야 합니다. 우리가 피하려는 것은 단순 import가 아니라 SDK 이벤트 모델이 Host의 사고방식이 되는 상황입니다.

**전환**  
그래서 다음 슬라이드에서는 Host가 어떤 언어만 쓰게 만들지 보겠습니다.

---

## 07. Host가 우리 언어만 쓰게 한다

**주장**  
Host는 SDK delegate가 아니라 command와 state만 다룹니다.

**화면**  
1. makeModule로 모듈을 만듭니다.
2. start와 execute로 행동을 보냅니다.
3. stateStream으로 화면을 그립니다.
4. 오른쪽 다이어그램에서 SDK는 Adapter 뒤에 있습니다.

**말할 내용**  
여기에는 KollusPlayerView, storage, SDK delegate가 없습니다. Host는 무엇을 할지만 command로 말하고, 결과는 state로 받습니다. SDK 차이는 EngineAdapter 안에서 흡수됩니다.

**전환**  
이 구조를 이해하려면 Core, Engine, Skin 세 축을 보면 됩니다.

---

## 08. 모듈의 핵심 구조

**주장**  
Core는 상태, Engine은 재생 방법, Skin은 화면 조립을 담당합니다.

**화면**  
1. 왼쪽 mental model은 Host, Core, Engine, Skin의 위치를 보여줍니다.
2. 오른쪽 카드에서 각 역할을 한 문장씩 확인합니다.
3. Host는 정책과 커스텀 UI를 주입하는 쪽입니다.

**말할 내용**  
Core는 재생 명령과 상태를 소유합니다. Engine은 AVPlayer, Kollus, 미래 SDK를 공통 계약으로 번역합니다. Skin은 고정 화면이 아니라 조립 가능한 UI를 제공합니다.

**전환**  
그럼 왜 SDK 의존을 줄여야 하는지 보겠습니다.

---

## 09. SDK는 사서 쓰되, 의존은 줄인다

**주장**  
SDK는 구매하되 앱 구조 전체가 SDK에 묶이면 안 됩니다.

**화면**  
1. BUY는 전문 SDK를 쓰는 이유입니다.
2. RISK는 SDK 정책, 가격, 품질 변화 가능성입니다.
3. CONTROL은 우리 구조를 command/state로 소유하자는 뜻입니다.

**말할 내용**  
VOD, DRM, 다운로드, 라이선스 서버는 직접 만드는 것보다 전문 SDK를 쓰는 편이 효율적입니다. 하지만 SDK가 바뀔 때 Host 화면 전체가 흔들리면 구매의 장점이 줄어듭니다. 그래서 SDK 타입은 Adapter 안에 가둡니다.

**전환**  
서비스별 차이는 주입 구조로 다룹니다.

---

## 10. POP와 Host 주입

**주장**  
서비스별 차이는 거대한 분기가 아니라 protocol과 주입으로 처리합니다.

**화면**  
1. 왼쪽 다이어그램은 Host가 policy, theme, block, engine을 넣는 구조입니다.
2. 오른쪽 코드는 policy와 blueprint를 조립하는 예시입니다.
3. configuration은 초기 정책을 모듈에 전달하는 지점입니다.

**말할 내용**  
서비스별 요구를 공통 코드의 if문으로 쌓으면 플레이어가 계속 커집니다. 이 모듈은 정해진 주입 지점으로 차이를 받습니다. 커스텀은 허용하지만 SDK나 내부 상태를 직접 만지는 방식은 막습니다.

**전환**  
Host 개발자가 실제로 알아야 하는 표면을 더 단순하게 보겠습니다.

---

## 11. Host가 알면 되는 것은 두 가지다

**주장**  
Host는 command를 보내고 state를 구독합니다.

**화면**  
1. 왼쪽 코드는 play, seek command를 보냅니다.
2. stateStream을 for await로 구독해 render합니다.
3. 오른쪽 리스트는 Host가 직접 소유하지 않는 것들을 정리합니다.

**말할 내용**  
Host는 “무엇을 하고 싶은지”만 command로 말합니다. 화면은 계산된 state를 받아 그립니다. SDK delegate, DRM callback, AVPlayer KVO는 Host가 직접 소유하지 않습니다.

**전환**  
이 command와 state가 런타임에서 어떻게 흐르는지 보겠습니다.

---

## 12. 런타임 흐름

**주장**  
명령과 상태는 한 방향으로 흐릅니다.

**화면**  
1. 위에서 아래로 흐름을 읽습니다.
2. Host/Skin이 command를 발행합니다.
3. Core가 정책과 capability를 보고 EngineAdapter에 위임합니다.
4. Reducer가 최종 PlaybackState를 결정합니다.

**말할 내용**  
단방향 흐름은 디버깅 기준을 명확하게 만듭니다. 마지막 command가 무엇인지, engine output이 무엇인지, reducer가 어떤 state를 만들었는지를 보면 됩니다.

**전환**  
이 흐름을 유지하려면 SDK가 들어오는 product 경계도 명확해야 합니다.

---

## 13. product 경계가 SDK 경계다

**주장**  
SDK 격리는 말이 아니라 SPM product 경계로 강제합니다.

**화면**  
1. 표에서 Core와 Skin은 SDK 의존이 없음을 봅니다.
2. Kollus product만 Kollus/PallyCon binary를 압니다.
3. 아래 Mermaid의 화살표는 의존 방향입니다. Core가 Engine 구현을 의존하지 않습니다.

**말할 내용**  
Core는 도메인, 상태, 엔진 계약만 압니다. Native와 Kollus engine은 Core/ShellSupport 쪽을 사용하고, vendor binary는 Kollus product 안에만 있습니다. 그래서 SDK 교체 영향 범위를 제한할 수 있습니다.

**전환**  
다음은 상태 소유권입니다.

---

## 14. 상태 소유권은 Core에 있다

**주장**  
SDK는 신호를 주고, 상태는 Core가 결정합니다.

**화면**  
1. 왼쪽 pseudo code는 callback이 reducer까지 들어가는 흐름입니다.
2. terminal 상태에서는 늦은 buffering이나 position 신호를 무시합니다.
3. 오른쪽 리스트는 왜 reducer가 필요한지 설명합니다.

**말할 내용**  
SDK callback 순서는 우리가 통제할 수 없습니다. 늦은 callback이 이미 끝난 상태를 되살리면 화면이 흔들립니다. 그래서 엔진은 신호만 내고, Core reducer가 현재 상태와 입력을 함께 보고 다음 상태를 결정합니다.

**전환**  
상태 다음으로는 기능 지원 차이를 보겠습니다.

---

## 15. 기능 차이는 capability로 처리한다

**주장**  
모든 엔진이 모든 기능을 지원하지 않아도 됩니다.

**화면**  
1. 위 공식은 hostPolicy, engineCapability, optionalProtocolSupport의 교집합입니다.
2. 세 카드가 각각 정책, 엔진 지원, UI 사전 게이트를 뜻합니다.
3. 지원 가능한 기능만 화면에 노출하는 것이 목적입니다.

**말할 내용**  
배속, 자막, 북마크, 다운로드, PiP는 엔진마다 지원 범위가 다를 수 있습니다. 사용자가 눌러보고 실패하게 두는 것이 아니라, 시작 시점에 가능한 기능만 Host와 Skin에 노출합니다.

**전환**  
이제 실제 조립 코드로 넘어가겠습니다. 먼저 Native engine입니다.

## 16. Native engine으로 개발하기

**주장**  
SDK가 필요 없는 화면은 Native engine만으로 가볍게 붙일 수 있습니다.

**화면**  
1. PlayerModuleWiring.makeModule에 AVPlayerAdapter를 넣습니다.
2. source는 .url(videoURL)입니다.
3. 재생은 동일하게 .play command로 보냅니다.

**말할 내용**  
일반 URL이나 HLS는 AVPlayerAdapter로 조립합니다. 모듈을 만든 뒤에는 start와 execute 흐름이 동일합니다. 미리보기, 광고, DRM 없는 콘텐츠에 적합한 경로입니다.

**전환**  
Kollus를 써야 하는 경우도 Host 사용 표면은 크게 달라지지 않습니다.

---

## 17. Kollus engine으로 개발하기

**주장**  
Kollus는 engine 안에 있고, Host는 같은 command/state 표면을 씁니다.

**화면**  
1. KollusEnvironment에 앱 키, 번들 ID, storage, DRM 설정을 넣습니다.
2. validate로 설정 오류를 일찍 잡습니다.
3. factory.makeModule 후 source는 .mediaKey(mediaContentKey)로 시작합니다.

**말할 내용**  
Kollus는 초기화 값이 필요하지만, 재생 표면은 Native와 같습니다. Core는 Kollus라는 이름을 모릅니다. mediaKey를 받고, 그 해석은 Kollus adapter가 담당합니다.

**전환**  
엔진만 바꾸는 것으로는 부족합니다. 서비스별 UI 조립도 필요합니다.

---

## 18. 서비스별 Skin 커스텀하기

**주장**  
공통 Core 위에 서비스별 Skin을 조립합니다.

**화면**  
1. 왼쪽 그림에서 slot은 자리입니다.
2. 오른쪽 코드에서 block은 그 자리에 들어가는 부품입니다.
3. visibleSlots는 layout mode별로 보일 자리를 정합니다.
4. theme은 서비스별 색과 스타일입니다.

**말할 내용**  
Skin은 고정 UI를 강제하지 않습니다. 같은 Core와 state를 쓰더라도 서비스별로 버튼 배치, 오버레이, 테마를 다르게 가져갈 수 있습니다. blueprint는 그 배치도 역할을 합니다.

**전환**  
이 구조가 실제로 어떻게 보이는지는 Example app에서 확인합니다.

## 19. Example app과 데모

**주장**  
데모는 기능 자랑이 아니라 구조 검증입니다.

**화면**  
1. HLS 재생은 Native engine과 state 흐름을 봅니다.
2. Kollus 실재생은 실기기에서 SDK 격리와 engine 교체를 봅니다.
3. DownloadCenter는 계약과 상태 모델만 확인합니다.
4. Skin 제스처는 UI 조립 가능성을 봅니다.

**말할 내용**  
여기서 볼 것은 버튼 하나가 동작한다는 사실이 아닙니다. 앞에서 말한 경계가 실제로 유지되는지 확인하는 것입니다. 특히 Kollus 실재생과 다운로드/오프라인은 유효한 콘텐츠와 실기기 QA에서 닫습니다.

**전환**  
경계가 유지되면 앞으로의 확장 방식도 달라집니다.

## 20. 미래 확장 시나리오

**주장**  
미래 변화는 Host 전체 수정이 아니라 경계 확장으로 흡수합니다.

**화면**  
1. 왼쪽 열은 앞으로 생길 수 있는 요구입니다.
2. 오른쪽 열은 대응할 확장 지점입니다.
3. SDK는 EngineAdapter, UI는 Skin block/theme, 기능 차이는 capability로 확장합니다.

**말할 내용**  
더 좋은 VOD SDK가 나오면 Host 화면을 다시 짜는 것이 아니라 새 EngineAdapter를 추가합니다. 서비스별 UI 요구는 Skin block이나 theme으로, 해외 서비스 요구는 policy와 localization 주입으로 대응합니다.

**전환**  
공통 모듈이 되려면 검증 경계도 함께 분리되어야 합니다.

---

## 21. 검증 전략과 남은 한계

**주장**  
공통 모듈은 코드 경계뿐 아니라 검증 경계도 가져야 합니다.

**화면**  
1. 왼쪽 표는 테스트 경계를 보여줍니다.
2. 오른쪽 카드는 아직 남은 한계를 보여줍니다.
3. Kollus 경로는 SDK 미지원 때문에 실기기 QA가 필요합니다.

**말할 내용**  
상태 전이는 reducer pure test로 봅니다. SDK 이벤트 매핑은 signal mapper test로 봅니다. Core나 Skin에 SDK 타입이 새지 않는지는 boundary test로 봅니다. Kollus 재생과 DRM, 다운로드, 백그라운드 오디오는 실기기 QA로 닫아야 합니다.

**전환**  
그래서 오늘 결정할 것은 세부 구현 전체가 아니라 다음 단계입니다.

## 22. 오늘 결정할 것

**주장**  
오늘은 구조 승인과 파일럿 착수를 결정하는 자리입니다.

**화면**  
1. D1은 Core/Engine/Skin 구조 승인입니다.
2. D2는 host 1개 화면부터 feature flag로 적용하는 파일럿입니다.
3. D3는 기존 UX 기준과 Host override 허용입니다.

**말할 내용**  
오늘 모든 세부 구현을 닫을 필요는 없습니다. 이 구조로 진행할지, 파일럿 범위를 어떻게 잡을지, 정책 차이를 어떤 원칙으로 다룰지 결정하면 다음 단계로 넘어갈 수 있습니다.

**전환**  
마지막으로 파일럿 계획을 정리하고 마무리하겠습니다.

## 23. 파일럿 계획과 마무리

**주장**  
이 모듈은 플레이어를 팀 공통 자산으로 만들기 위한 첫 단계입니다.

**화면**  
1. 큰 문장 “여러 서비스가 다시 만들지 않도록”을 먼저 읽습니다.
2. 아래 단계는 Example app, host 1개 화면, 실기기 QA, 확장 여부 결정입니다.
3. 마지막 버튼은 오늘 요청하는 구조 승인과 파일럿 착수입니다.

**말할 내용**  
처음 문제로 돌아가면, 목표는 멋있는 플레이어 하나를 더 만드는 것이 아닙니다. 여러 서비스가 같은 플레이어 문제를 다시 풀지 않도록 공통 재생 구조를 팀 자산으로 만드는 것입니다. 파일럿은 되돌릴 수 있게 작게 시작합니다.

**전환**  
이 구조로 파일럿을 시작할 수 있을지 결정하고 싶습니다.
