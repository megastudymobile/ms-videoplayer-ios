problem · 레거시 플레이어

god object

SDK 콜백 26개를 화면 클래스가 직접 처리. UI · 재생 로직 · 서비스 용어가 한 파일에서 성장.

상태 분산 → 역전 버그

영상이 끝난 뒤 버퍼링 해제 콜백이 도착하면 재생 상태가 부활. 콜백 순서를 막는 단일 지점 부재.

테스트 불가

SDK 없이 상태 전이를 검증할 방법이 없음 — 모든 검증이 실기기 수동 QA.

벤더 강결합

DRM SDK 타입이 화면 코드까지 침투. 엔진 교체 = 화면 전면 수정. 어느 플랫폼이든 같은 함정.

design principle — platform neutral

화면은 의도만,
엔진은 방법만,
상태는 순수 함수가.

화면은 "재생해줘 · 30초로 이동해줘"라는 의도만 발행합니다.
SDK 호출 방법은 엔진 어댑터가, 다음 상태 계산은 순수 함수가 맡습니다.
iOS 구현이지만, 패턴 자체는 플랫폼과 무관합니다.

skin

재사용 UI

블록 단위로 조립하는 플레이어 UI. 엔진을 모르고, 액션·상태 계약으로만 통신.

강제 수단: 경계 위반(벤더 import·서비스 용어 침투)은 자동 테스트가 빌드에서 차단합니다.

core idea · state machine

상태를 움직이는 신호는 단 7종.

signals → reducer(state, signal) → next state

prepared · playStarted · pauseStarted
bufferingChanged · positionChanged
seeking · stopped / failed

  • 순수 함수 — SDK·기기 없이 일반 단위 테스트로 전이 규칙 전체 검증.
  • 잘못된 전이 차단 — 종료 후 도착한 버퍼링 해제 신호가 재생 상태를 부활시키지 못함.
  • 벤더가 달라도 동일 — 어떤 엔진이든 신호 7종으로 정규화되면 같은 규칙 적용.
proof

레거시 문제 → 설계 요소 매핑.

레거시 문제해결

상태 역전 버그reducer가 전이 규칙으로 차단 — 종료 상태는 되돌릴 수 없음
실기기에서만 테스트순수 로직은 CI에서 즉시 검증 — 232개 테스트 통과
벤더 강결합도메인·UI는 SDK 무지 — 위반은 자동 테스트가 차단
만료 콘텐츠 사후 감지재생 전 DRM 사전 검증 — 시도 전에 실패 알림

usage · concept

사용하는 쪽은 네 단계가 전부.

개념 — 플랫폼 무관 
// 1) 조립 — 엔진만 선택
module = makeModule(engine: drmEngine | nativeEngine)

// 2) 상태 구독
observe module.stateStream → render(state)

// 3) 의도 발행
module.start(source), module.execute(.play)

// 4) 렌더 표면 부착
module.bind(surface: view)
  • 엔진을 바꿔도 1·3·4는 동일 — 화면 코드는 벤더를 구분하지 않음.
  • 모든 명령은 실패 가능 — 에러가 타입으로 노출, 숨겨지지 않음.
  • 다운로드·DRM도 중립 계약 — 벤더 타입은 조립 지점에서만 등장.

순수 로직 테스트

재현 가능한 SDK 패키징

벤더 산출물 원본 불변 + 스크립트 재생성 + checksum 검증.
