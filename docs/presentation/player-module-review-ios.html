Script 00. 플레이어 공통 재생 모듈

말할 내용: 오늘 발표의 주제는 서비스별로 다시 만들던 플레이어를 공통 재생 모듈로 바꾸자는 제안입니다.
순서는 문제와 비용, 목표 구조, 적용 방식, 검증과 결정 순서로 보겠습니다.
먼저 왜 지금 이 논의를 해야 하는지부터 시작하겠습니다.

전환: 먼저 현재 반복 구현 문제가 왜 생겼는지 보겠습니다.

화면:
1. 큰 제목으로 오늘 주제가 플레이어 공통 재생 모듈임을 말합니다.
2. 배경 mesh는 기존 도입 슬라이드의 분위기를 커버로 옮긴 것입니다.
3. 하단 발표 순서를 문제, 구조, 적용, 결정 순서로 안내합니다.

why now · shared player module

지금 필요한 것은
플레이어 공통화

여러 서비스가 각자 다시 붙이던 플레이어를, 공통 재생 모듈로 재사용 가능하게 만들기 위해서입니다.

HIGH고등 서비스오래된 플레이어 코드 유지
KIDS유초등네이티브 플레이어 재구현
GLOBAL해외사업부서비스별 요구에 맞춰 재구현
GOAL공통 모듈한 번 만들고 여러 서비스에서 재사용

플레이어는 서비스별로 다시 만들 영역이 아니라, 팀 공통 자산이 되어야 합니다.

Script 01. 지금 필요한 것은 플레이어 공통화

말할 내용: 지금 모듈화가 필요한 이유는 여러 서비스가 같은 재생 문제를 반복해서 풀고 있기 때문입니다.
재생, 제스처, 자막, 배속, DRM, 다운로드는 대부분 공통 문제입니다.
그래서 출발점은 “여러 서비스가 같은 문제를 다시 풀지 않게 하자”입니다.

전환: 그 반복 구현이 실제로 어떤 비용으로 남는지 보겠습니다.

화면:
1. 고등, 유초등, 해외사업부 카드를 차례로 짚습니다.
2. 마지막 GOAL 카드에서 공통 모듈이 목적임을 말합니다.
3. 하단 takeaway로 결론을 고정합니다.

Script 02. 반복 구현이 키우는 비용

말할 내용: 반복 구현이 키우는 비용은 기능 개발비보다 유지보수와 검증 비용입니다.
특히 Kollus 실재생 경로는 시뮬레이터에서 닫기 어렵기 때문에 실기기 검증이 필요합니다.
공통 경계가 없으면 SDK 버전 대응과 사용자 경험 차이도 계속 서비스별 비용으로 남습니다.

전환: 이 비용이 실제 코드 표면에 어떻게 남아 있는지 보겠습니다.

화면:
1. UX 카드는 제스처, 자막, 배속 반복 구현을 뜻합니다.
2. KOLLUS 카드는 실재생 경로가 실기기 검증 대상이라는 제약입니다.
3. SDK와 PRODUCT 카드는 버전 대응과 경험 분산 비용입니다.

Script 03. 코드와 QA 표면에 남은 반복 비용

말할 내용: 반복 비용은 SDK 접점, delegate, storage, 다운로드 API가 여러 곳에 흩어진 형태로 남아 있습니다.
이 숫자는 “느낌상 복잡하다”가 아니라 코드 표면에 이미 비용이 남아 있다는 근거입니다.
흩어진 변경과 검증을 줄이려면 공통 경계가 필요합니다.

전환: 이제 목표 구조를 보겠습니다.

화면:
1. 72는 host 레거시 플레이어 영역 기준 Objective-C 파일 수입니다.
2. 82는 같은 기준의 Kollus 관련 SDK 접점 검색 결과입니다.
3. QA 카드는 Kollus 실재생/DRM/다운로드가 실기기 검증 대상임을 뜻합니다.

Script 04. 여러 서비스가 재사용하는 목표 구조

말할 내용: 재사용 가능한 구조는 공통 재생 경계와 서비스별 주입 지점을 나누는 구조입니다.
재생 명령, 상태, 정책 협상, 엔진 계약은 공통화하고, 서비스별 UI나 정책은 주입합니다.
모든 서비스가 같은 화면을 쓰자는 뜻은 아닙니다.

전환: 구조 이야기에 들어가기 전에 용어를 맞추고 가겠습니다.

화면:
1. 왼쪽 묶음은 이 모듈을 소비하는 서비스 Host입니다.
2. 가운데 묶음은 여러 서비스가 재사용할 공통 재생 모듈입니다.
3. 오른쪽 묶음은 서비스별로 달라질 Host 주입점입니다.
4. 화살표는 Host가 공통 모듈을 재사용하고, 필요한 차이만 주입한다는 흐름입니다.

shared vocabulary

오늘의 핵심 용어

CORE

상태 소유자

명령을 받고 다음 상태를 결정합니다.

ENGINE

공통 언어

Host가 보내고 구독하는 사용 표면입니다.

이후 슬라이드는 이 다섯 단어가 어디에서 만나고 분리되는지를 설명합니다.

Script 05. 오늘의 핵심 용어

말할 내용: 오늘 잡고 갈 단어는 Host, Core, Engine, Skin, Command/State입니다.
이 다섯 단어가 어디에서 만나고 어디에서 분리되는지만 따라가면 됩니다.
모듈화가 낯설어도 전체 흐름을 놓치지 않을 수 있습니다.

전환: 이제 단순 wrapper와 공통 모듈의 차이를 코드로 보겠습니다.

화면:
1. Host는 서비스 앱입니다.
2. Core는 상태 소유자입니다.
3. Engine은 실제 SDK를 호출하는 번역기입니다.
4. Skin은 화면 조립이고, Command/State는 공통 언어입니다.

TYPE LEAK

SDK 타입이 화면 책임이 됩니다.

`KollusPlayerView`, storage, delegate가 Host 계층의 사고방식이 됩니다.

STATE LEAK

콜백이 화면 상태를 움직입니다.

play, pause, buffering callback 순서를 화면이 직접 해석합니다.

RULE LEAK

SDK 예외 규칙이 서비스 코드에 쌓입니다.

타임아웃, 다운로드, DRM, 재시도 규칙이 wrapper 밖으로 번집니다.

이 상태에서 SDK만 감싸면 호출 위치는 줄어도, Host가 SDK의 생각법을 계속 따라가게 됩니다.

Script 06. 얇은 Wrapper가 새는 지점

말할 내용: 얇은 Wrapper는 SDK 타입, delegate, callback 순서가 Host까지 올라오는 지점에서 샙니다.
이런 구조에서는 wrapper를 만들어도 Host가 여전히 SDK의 조립 규칙을 알아야 합니다.
우리가 피하려는 것은 SDK 이벤트 모델이 Host의 사고방식이 되는 상황입니다.

전환: 그래서 다음 슬라이드에서는 Host가 어떤 언어만 쓰게 만들지 보겠습니다.

화면:
1. 노란색 KollusPlayerView를 봅니다. 화면 계층이 SDK 타입을 직접 압니다.
2. 분홍색 setStorage, setDelegate를 봅니다. SDK 조립 규칙도 화면에 올라옵니다.
3. 마지막 상태 enum을 봅니다. SDK 콜백이 화면 상태를 직접 움직입니다.

Script 07. Host가 쓰는 언어: command와 state

말할 내용: Host가 쓰게 만들 언어는 PlaybackCommand와 PlaybackState입니다.
여기에는 KollusPlayerView, storage, SDK delegate가 없습니다.
Host는 무엇을 할지만 command로 말하고, 결과는 state로 받습니다.

전환: 이 구조를 이해하려면 Core, Engine, Skin 세 축을 보면 됩니다.

화면:
1. makeModule로 모듈을 만듭니다.
2. start와 execute로 행동을 보냅니다.
3. stateStream으로 화면을 그립니다.
4. 오른쪽 다이어그램에서 SDK는 Adapter 뒤에 있습니다.

Script 08. Core · Engine · Skin 세 축

말할 내용: 이 모듈의 세 축은 Core, Engine, Skin입니다.
Core는 재생 명령과 상태를 소유하고, Engine은 AVPlayer, Kollus, 미래 SDK를 공통 계약으로 번역합니다.
Skin은 고정 화면이 아니라 조립 가능한 UI를 제공합니다.

전환: 그럼 왜 SDK 의존을 줄여야 하는지 보겠습니다.

화면:
1. mental model을 위에서 아래로 읽습니다.
2. Host는 정책과 커스텀 UI를 주입하고 command를 보냅니다.
3. Core가 상태를 소유하고, Engine과 Skin은 각각 재생 방법과 화면 조립을 담당합니다.

sdk strategy

SDK는 구매하고, 구조는 소유합니다

BUY

VOD/DRM은 구매

트랜스코딩, CDN, DRM, 라이선스 서버는 전문 SDK를 쓰는 편이 효율적입니다.

RISK

의존 리스크

SDK 정책, 가격, 품질, 지원 범위는 언제든 바뀔 수 있습니다.

CONTROL

구조는 우리가 소유

SDK 타입은 adapter 안에 가두고 Host는 command/state만 씁니다.

Script 09. SDK는 구매하고, 구조는 소유합니다

말할 내용: SDK 의존도를 줄여야 하는 이유는 구매한 SDK의 변화가 앱 구조 전체를 흔들 수 있기 때문입니다.
VOD, DRM, 다운로드, 라이선스 서버는 전문 SDK를 쓰는 편이 효율적입니다.
하지만 SDK 타입은 Adapter 안에 가둬야 구매의 장점을 유지할 수 있습니다.

전환: 서비스별 차이는 주입 구조로 다룹니다.

화면:
1. BUY는 전문 SDK를 쓰는 이유입니다.
2. RISK는 SDK 정책, 가격, 품질 변화 가능성입니다.
3. CONTROL은 우리 구조를 command/state로 소유하자는 뜻입니다.

Script 10. 서비스별 차이는 주입 지점으로 분리

말할 내용: 서비스별 차이는 공통 코드의 if문이 아니라 정해진 protocol과 주입 지점에 넣어야 합니다.
이 모듈은 policy, theme, block, engine 같은 경계로 차이를 받습니다.
커스텀은 허용하지만 SDK나 내부 상태를 직접 만지는 방식은 막습니다.

전환: Host 개발자가 실제로 알아야 하는 표면을 더 단순하게 보겠습니다.

화면:
1. 왼쪽 다이어그램은 Host가 policy, theme, block, engine을 넣는 구조입니다.
2. 오른쪽 코드는 policy와 blueprint를 조립하는 예시입니다.
3. configuration은 초기 정책을 모듈에 전달하는 지점입니다.

host contract

Host 계약은 command와 state

HostViewController.swift 
try await module.core.execute(command: .play)
try await module.core.execute(command: .seek(to: time))

for await state in module.core.stateStream {
    render(state)
}
  • Host는 무엇을 할지 command로 말합니다.
  • 모듈은 어떻게 재생할지 engine으로 처리합니다.
  • Host는 계산된 state만 받아 화면을 그립니다.
  • SDK delegate, DRM callback, AVPlayer 상태 전이를 직접 소유하지 않습니다.

여러 서비스가 재사용할 수 있는 가장 중요한 표면은 command와 state입니다.

Script 11. Host 계약은 command와 state

말할 내용: Host가 알아야 할 표면은 command를 보내고 state를 구독하는 두 가지입니다.
Host는 “무엇을 하고 싶은지”만 command로 말하고, 화면은 계산된 state를 받아 그립니다.
SDK delegate, DRM callback, AVPlayer KVO는 Host가 직접 소유하지 않습니다.

전환: 이 command와 state가 런타임에서 어떻게 흐르는지 보겠습니다.

화면:
1. 왼쪽 코드는 play, seek command를 보냅니다.
2. stateStream을 for await로 구독해 render합니다.
3. 오른쪽 리스트는 Host가 직접 소유하지 않는 것들을 정리합니다.

Script 12. 명령은 내려가고 상태는 돌아옵니다

말할 내용: 명령과 상태는 Skin에서 Core를 거쳐 Engine으로 내려가고, 출력은 Reducer를 거쳐 state로 돌아옵니다.

그래서 디버깅할 때 action, command, engine output, reducer state를 순서대로 보면 됩니다.

전환: 이 흐름을 유지하려면 SDK가 들어오는 product 경계도 명확해야 합니다.

화면:
1. 위에서 아래로 흐름을 읽습니다.
2. Skin은 PlayerSkinAction을 내보내고 Host가 PlaybackCommand로 변환합니다.
3. Core가 정책과 capability를 보고 EngineAdapter에 위임합니다.
4. Reducer가 최종 PlaybackState를 결정합니다.

Script 13. SDK는 Engine product 안에만 둡니다

말할 내용: SDK는 Core나 Skin까지 들어오면 안 되고, 필요한 engine product 안에만 머물러야 합니다.
Core는 도메인, 상태, 엔진 계약만 알고, ShellSupport는 wiring과 render surface를 담당합니다.
vendor binary는 Kollus product 안에만 있습니다.

전환: 다음은 상태 소유권입니다.

화면:
1. 표에서 Core, ShellSupport, Skin은 SDK 의존이 없음을 봅니다.
2. Kollus product만 Kollus/PallyCon binary를 압니다.
3. 아래 Mermaid의 화살표는 의존 방향입니다. Core가 Engine 구현을 의존하지 않습니다.

state ownership

상태 결정은 Core reducer가 소유

state transition pseudo code 
SDK callback
  -> PlayerEngineOutput
  -> PlaybackStateInput
  -> reducer(currentState, input)
  -> next PlaybackState

reduce(input, state):
    if state.status is terminal:
        ignore late buffering / position signal

    return transition(state, input)
  • SDK 콜백 순서는 우리가 통제할 수 없습니다.
  • 늦은 buffering 콜백이 finished 상태를 되살리면 안 됩니다.
  • 그래서 상태 전이는 SDK 밖의 reducer가 소유합니다.
  • 이 로직은 vendor SDK 없이 pure test로 검증합니다.

Script 14. 상태 결정은 Core reducer가 소유

말할 내용: SDK 콜백은 Engine이 신호로 바꾸고, 최종 상태 결정은 Core reducer가 해야 합니다.
SDK callback 순서는 우리가 통제할 수 없고, 늦은 callback이 이미 끝난 상태를 되살리면 화면이 흔들립니다.
그래서 Core가 현재 상태와 입력을 함께 보고 다음 상태를 결정합니다.

전환: 상태 다음으로는 기능 지원 차이를 보겠습니다.

화면:
1. 왼쪽 pseudo code는 callback이 reducer까지 들어가는 흐름입니다.
2. terminal 상태에서는 늦은 buffering이나 position 신호를 무시합니다.
3. 오른쪽 리스트는 왜 reducer가 필요한지 설명합니다.

Script 15. 기능 차이는 availability와 capability로 관리

말할 내용: 엔진마다 다른 기능 지원은 optional protocol, capability, Host policy를 나눠서 다룹니다.
배속, 자막, 북마크, PiP 같은 버튼 노출은 엔진이 어떤 optional protocol을 채택했는지로 결정합니다.
다운로드는 PlayerDownloadCenter 계약으로 별도 노출합니다.

전환: 이제 실제 조립 코드로 넘어가겠습니다. 먼저 Native engine입니다.

화면:
1. 위 공식은 현재 코드 기준의 두 축입니다.
2. UI 가용 기능은 엔진 optional protocol 채택으로 산출합니다.
3. engineCapabilities는 백그라운드 유지나 command-origin 같은 런타임 정책 조정에 씁니다.

일반 URL/HLS는 Native engine으로 조립

SDK가 필요 없는 미리보기, 광고, DRM 없는 영상은 Native engine만으로 가볍게 붙일 수 있습니다.

Script 16. 일반 URL/HLS는 Native engine으로 조립

말할 내용: 일반 URL이나 HLS는 AVPlayerAdapter를 넣어 Native engine 경로로 조립합니다.
모듈을 만든 뒤에는 start와 execute 흐름이 동일합니다.
미리보기, 광고, DRM 없는 콘텐츠에 적합한 경로입니다.

전환: Kollus를 써야 하는 경우도 Host 사용 표면은 크게 달라지지 않습니다.

화면:
1. PlayerModuleWiring.makeModule에 AVPlayerAdapter를 넣습니다.
2. source는 .url(videoURL)입니다.
3. 재생은 동일하게 .play command로 보냅니다.

develop · kollus engine

Kollus는 전용 Engine 안에서 조립

KollusPlayerAssembly.swift 
let environment = KollusEnvironment(
    applicationKey: key,

Kollus는 engine 안에 있고, Host는 같은 command/state 표면을 씁니다.

Script 17. Kollus는 전용 Engine 안에서 조립

말할 내용: Kollus는 environment와 mediaKey 해석 지점에서만 다르게 조립합니다.
초기화 값은 필요하지만, Host가 보는 재생 표면은 Native와 같습니다.
Core는 Kollus라는 이름을 모르고, 해석은 Kollus adapter가 담당합니다.

전환: 엔진만 바꾸는 것으로는 부족합니다. 서비스별 UI 조립도 필요합니다.

화면:
1. KollusEnvironment에 앱 키, 번들 ID, storage, DRM 설정을 넣습니다.
2. validate로 설정 오류를 일찍 잡습니다.
3. factory.makeModule 후 source는 .mediaKey(mediaContentKey)로 시작합니다.

화면:
1. HLS 재생은 Native engine과 state 흐름을 봅니다.
2. Kollus 실재생은 실기기에서 SDK 격리와 engine 교체를 봅니다.
3. DownloadCenter는 계약과 상태 모델만 확인합니다.
4. Skin 제스처는 UI 조립 가능성을 봅니다.
