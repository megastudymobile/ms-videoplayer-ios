Script 00. 플레이어 공통 재생 모듈

말할 내용: 오늘 발표의 주제는 서비스별로 다시 만들던 플레이어를 공통 재생 모듈로 바꾸자는 제안입니다.
순서는 문제와 비용, 목표 구조, 적용 방식, 검증과 결정 순서로 보겠습니다.
오늘 마지막에는 구조 승인, 파일럿 범위, 정책 원칙 세 가지 결정을 요청드립니다.
먼저 왜 지금 이 논의를 해야 하는지부터 시작하겠습니다.

전환: 먼저 현재 반복 구현 문제가 왜 생겼는지 보겠습니다.

화면:
1. 큰 제목으로 오늘 주제가 플레이어 공통 재생 모듈임을 말합니다.
2. 배경 mesh는 기존 도입 슬라이드의 분위기를 커버로 옮긴 것입니다.
3. 하단 발표 순서를 문제, 구조, 적용, 결정 순서로 안내합니다.

Script 01. 지금 필요한 것은 플레이어 공통화

말할 내용: 지금 모듈화가 필요한 이유는 여러 서비스가 같은 재생 문제를 반복해서 풀고 있기 때문입니다.

화면:
1. 고등, 유초등, 해외사업부 카드를 차례로 짚습니다.
2. 마지막 GOAL 카드에서 공통 모듈이 목적임을 말합니다.
3. 하단 takeaway로 결론을 고정합니다.

Script 02. 반복 비용은 코드와 QA 표면에 남아 있습니다

말할 내용: 반복 구현이 키우는 비용은 기능 개발비보다 유지보수와 검증 비용입니다.
같은 재생 버그라도 서비스별 코드가 나뉘어 있으면 수정과 QA가 서비스 수만큼 반복됩니다.
현재 레거시 영역에는 Objective-C 파일과 SDK 접점이 넓게 남아 있고, Kollus 실재생과 DRM, 다운로드 경로는 실기기 검증 대상입니다.

전환: 그래서 단순히 SDK를 얇게 감싸는 wrapper만으로 충분한지 보겠습니다.

화면:
1. 위 카드 세 개는 사용자 경험, 코드 표면, SDK 접점이라는 반복 비용의 근거입니다.
2. 하단 흐름은 플레이어 버그 1건이 서비스별 수정과 실기기 QA로 분기되는 모습을 보여줍니다.
3. 오른쪽 공통 모듈 박스는 수정 1회, 검증 1회로 줄이는 목표입니다.

Script 03. 오늘의 핵심 용어

말할 내용: 오늘 잡고 갈 단어는 Host, Core, Engine, Skin, Command/State입니다.
이 다섯 단어가 어디에서 만나고 어디에서 분리되는지만 따라가면 됩니다.
모듈화가 낯설어도 전체 흐름을 놓치지 않을 수 있습니다.

전환: 이 용어로 여러 서비스가 재사용할 목표 구조를 보겠습니다.

화면:
1. Host는 서비스 앱입니다.
2. Core는 상태 소유자입니다.
3. Engine은 실제 SDK를 호출하는 번역기입니다.
4. Skin은 화면 조립이고, Command/State는 공통 언어입니다.

TYPE LEAK

SDK 타입이 화면 책임이 됩니다.

`KollusPlayerView`, storage, delegate가 Host 계층의 사고방식이 됩니다.

STATE LEAK

콜백이 화면 상태를 움직입니다.

play, pause, buffering callback 순서를 화면이 직접 해석합니다.

RULE LEAK

SDK 예외 규칙이 서비스 코드에 쌓입니다.

타임아웃, 다운로드, DRM, 재시도 규칙이 wrapper 밖으로 번집니다.

이 상태에서 SDK만 감싸면 호출 위치는 줄어도, Host가 SDK의 생각법을 계속 따라가게 됩니다.

Script 04. 얇은 Wrapper가 새는 지점

말할 내용: 얇은 Wrapper는 SDK 타입, delegate, callback 순서가 Host까지 올라오는 지점에서 샙니다.
이런 구조에서는 wrapper를 만들어도 Host가 여전히 SDK의 조립 규칙을 알아야 합니다.
우리가 피하려는 것은 SDK 이벤트 모델이 Host의 사고방식이 되는 상황입니다.

전환: 이런 누수를 막는 구조를 Core, Engine, Skin 세 축으로 보겠습니다.

화면:
1. 노란색 KollusPlayerView를 봅니다. 화면 계층이 SDK 타입을 직접 압니다.
2. 분홍색 setStorage, setDelegate를 봅니다. SDK 조립 규칙도 화면에 올라옵니다.
3. 마지막 상태 enum을 봅니다. SDK 콜백이 화면 상태를 직접 움직입니다.

Script 05. Core · Engine · Skin으로 재사용 경계를 나눕니다

말할 내용: 얇은 wrapper의 누수를 막으려면 공통 재생 구조와 서비스별 주입 지점을 나눠야 합니다.
Core는 재생 명령과 상태를 소유하고, Engine은 AVPlayer, Kollus, 미래 SDK를 공통 계약으로 번역합니다.
Skin은 고정 화면이 아니라 조립 가능한 UI를 제공합니다.
모든 서비스가 같은 화면을 쓰자는 뜻이 아니라, 같은 재생 구조 위에 필요한 차이를 주입하자는 뜻입니다.

전환: 이제 이 세 축 위에서 Host가 실제로 어떤 언어만 쓰는지 보겠습니다.

화면:
1. Host는 서비스별 정책, engine, skin block을 주입합니다.
2. Core는 명령과 상태, reducer를 공통 책임으로 소유합니다.
3. Engine과 Skin은 각각 재생 방법과 화면 조립을 담당합니다.

Script 08. Kollus SDK는 여기까지만 들어옵니다

말할 내용: 이 슬라이드에서 볼 것은 의존성 전체가 아니라 SDK가 멈추는 위치입니다.

전환: 이제 SDK 신호가 공통 상태로 바뀐 뒤, 최종 상태를 누가 결정하는지 보겠습니다.

화면:
1. 왼쪽은 SDK 비의존 영역입니다.
2. 가운데 Engine Contract가 경계입니다.
3. 오른쪽만 vendor binary를 직접 압니다.
4. 기억할 문장은 하나입니다. SDK는 Kollus Engine 안에서 끝납니다.

Script 09. 상태 결정은 Core reducer가 소유

말할 내용: SDK 콜백은 Engine이 신호로 바꾸고, 최종 상태 결정은 Core reducer가 해야 합니다.
같은 buffering 입력이라도 playing에서는 상태 전이, paused에서는 플래그 변경, finished에서는 무시처럼 현재 상태가 결정에 참여합니다.
일시정지 중 buffering 종료를 무조건 playing으로 바꾸면 일시정지가 재생으로 둔갑하기 때문에 reducer가 필요합니다.

전환: 이 원칙이 담긴 엔진 계약을 직접 보겠습니다.

화면:
1. 왼쪽 pseudo code는 reducer가 currentState와 input을 함께 본다는 점을 보여줍니다.
2. 오른쪽 표는 같은 buffering 입력도 현재 상태에 따라 결정이 달라지는 예입니다.
3. 아래 타임라인은 finished 뒤 늦은 buffering이 reducer에서 차단되어 Host 상태가 흔들리지 않는 흐름입니다.

engine contract

엔진 계약은 명령 5개와 출력 1개

PlayerPlaybackEngine.swift · 모든 엔진의 공통 계약 
public protocol PlayerPlaybackEngine: Actor {
    nonisolated static var runtimeTraits: EngineRuntimeTraits { get }

    func prepare(source: PlaybackSource) async throws
    func play() async throws
    func pause() async throws
    func seek(to time: TimeInterval) async throws
    func stop(reason: PlayerStopReason) async throws

    // 엔진의 유일한 출력 — Core가 소비해 상태를 만든다
    var outputStream: AsyncStream<PlayerEngineOutput> { get }
}
  • runtimeTraits — 엔진 동작 특성을 선언하고 Core가 정책 협상에 사용합니다.
  • prepare(source) — URL이든 mediaKey든 source를 해석해 재생을 준비합니다.
  • play() / pause() — 재생 제어. 성공해도 상태는 outputStream 신호로 확정됩니다.
  • seek(to) — 위치 이동 명령입니다.
  • stop(reason) — userClosed, replacedSource, appLifecycle, finished를 구분합니다.
  • outputStream — 엔진의 유일한 출력. Core가 소비해 상태를 만듭니다.

명령은 async throws로 실패를 드러내고, 배속·자막·PiP는 ability protocol로 확장합니다.

Script 10. 엔진 계약은 명령 5개와 출력 1개

말할 내용: 엔진 교체가 현실이 되려면 기본 계약이 단순해야 합니다.
runtimeTraits는 엔진 동작 특성이고, prepare는 source를 해석해 재생 준비를 합니다.
play, pause, seek, stop은 모두 async throws 명령이고, 성공해도 최종 상태 확정은 outputStream 신호를 Core가 해석해서 만듭니다.
Actor 격리는 엔진 내부 상태 변경 순서를 지키고, 배속·자막·PiP 같은 기능은 다음 슬라이드의 ability protocol로 확장합니다.

전환: 이 기본 계약 위에서 엔진별 기능 차이를 어떻게 다루는지 보겠습니다.

화면:
1. 코드는 runtimeTraits, prepare, play/pause, seek, stop, outputStream 순서로 읽습니다.
2. 오른쪽 리스트는 각 항목이 맡는 역할입니다.
3. async throws는 실패를 숨기지 않는 계약이고, outputStream은 엔진의 유일한 단방향 출력입니다.
4. 마지막 takeaway는 기본 계약 밖의 기능은 ability protocol로 확장한다는 뜻입니다.

availability · runtime traits

버튼 노출과 실제 동작은 따로 판단합니다

Host 정책은 “허용하고 싶다”이고, runtime traits는 “이 엔진이 실제로 버틸 수 있다”입니다. 둘이 충돌하면 Core는 런타임에서 더 안전한 effective policy를 씁니다.

정리하면 availability는 화면 노출, runtime traits는 “노출된 뒤 안전하게 동작시키는 방식”을 결정합니다. 노출 조건은 블록이 requiredFeatures로 직접 신고합니다.

Script 11. 기능 차이는 availability와 runtime traits로 관리

말할 내용: 여기서는 기능 차이를 두 질문으로 나눕니다.
첫 번째는 “버튼을 보여줄까?”입니다. 배속 버튼은 엔진이 EnginePlaybackRateAbility를 채택하고 Host 정책도 배속을 허용해야 보입니다. 둘 중 하나라도 아니면 Skin에서 아예 노출하지 않습니다.
이 노출 조건은 화면 코드가 if로 들고 있지 않습니다. 블록이 requiredFeatures로 자기 조건을 신고하고, 엔진이 확정되면 skin이 일괄 게이팅합니다. 엔진이 ability 채택으로 능력을 신고하듯, 블록은 requiredFeatures로 수요를 신고하는 대칭 구조입니다.
두 번째는 “보여준 뒤 실제로 안전하게 동작할까?”입니다. 예를 들어 Host가 백그라운드 재생을 허용해도, 엔진이 surface 없이 재생을 유지할 수 없으면 Core는 런타임 effectivePolicy에서 백그라운드 재생을 false로 낮춥니다. 그리고 policyDowngraded 이벤트로 Host에 알려줍니다.
즉 하향 조정은 Host 설정을 몰래 바꾸는 게 아니라, 현재 엔진으로 지킬 수 없는 약속을 런타임에서 안전한 동작으로 줄이는 것입니다.

전환: 이제 이 기능 게이트를 서비스별 UI 조립에서 어떻게 쓰는지 보겠습니다.

화면:
1. 위 레인은 배속 버튼 노출 조건입니다. ability 채택과 Host 정책이 모두 필요합니다.
2. 아래 레인은 백그라운드 재생 예시입니다. Host는 허용했지만 EngineRuntimeTraits의 surface.continuesWithoutSurface가 false면 Core가 effectivePolicy를 false로 낮춥니다.
3. 마지막 문장은 이 슬라이드의 결론입니다. availability는 화면 노출, runtime traits는 노출된 뒤의 안전한 동작 방식을 결정합니다.

Script 13. 데모에서 확인할 구조

말할 내용: 데모에서 확인할 것은 기능 하나가 아니라 앞에서 말한 경계가 실제로 유지되는지입니다.
HLS, Kollus, DownloadCenter, Skin 제스처가 같은 command/state 표면으로 연결되는지 봅니다.
다만 현재 데모앱에는 실제 다운로드에 사용할 Kollus URL이 없어서, DownloadCenter는 계약과 상태 모델, URL-missing 처리까지만 확인합니다. 실다운로드 E2E는 host/API가 유효한 다운로드 URL을 제공한 뒤 닫습니다.

전환: 경계가 유지되면 앞으로의 확장 방식도 달라집니다.

화면:
1. HLS 재생은 Native engine과 state 흐름을 봅니다.
2. Kollus 실재생은 실기기에서 SDK 격리와 engine 교체를 봅니다.
3. DownloadCenter는 현재 URL 부재로 계약, 상태 모델, URL-missing 처리까지만 확인합니다.
4. Skin 제스처는 UI 조립 가능성을 봅니다.

Script 14. 미래 확장은 정해진 경계로 흡수

말할 내용: 앞으로의 확장은 Host 전체 수정이 아니라 정해진 경계의 확장으로 처리합니다.
더 좋은 VOD SDK가 나오면 새 EngineAdapter를 추가하고, UI 요구는 Skin block이나 theme으로 받습니다.
다운로드 정책과 해외 서비스 요구는 DownloadCenter, policy, localization 주입으로 대응합니다.
새 기능 추가는 PlayerFeature에 case 하나를 더하는 것으로 시작합니다. 지원 검사와 정책 판단이 default 없는 switch라서, 갱신해야 할 곳을 전부 컴파일 에러가 알려줍니다. 체크리스트를 사람이 기억하지 않습니다.

전환: 공통 모듈이 되려면 검증 경계도 함께 분리되어야 합니다.

화면:
1. 왼쪽 열은 앞으로 생길 수 있는 요구입니다.
2. 오른쪽 열은 대응할 확장 지점입니다.
3. SDK는 EngineAdapter, UI는 Skin block/theme, 기능 차이는 PlayerFeature case와 ability protocol로 확장합니다. exhaustive switch 덕에 누락은 컴파일 에러로 드러납니다.
